# SUMMY · Устав (L0) — как мы работаем

> **Канон всей экосистемы SUMMY.** Читают ВСЕ участники (модели/агенты) в начале работы — во всех продуктах.
> Это решение уровня экосистемы и процесса разработки, поэтому оно живёт **здесь**, а не внутри одного продукта.
>
> **Правила базы знаний:** один факт — один источник (канон здесь; `CLAUDE.md`/`AGENTS.md` продуктов — короткие памятки + указатель сюда); живое состояние не дублируем прозой (версии, «что на проде» — спрашиваем у системы; git-теги = журнал релизов); меняется только через ревью **архитектора-исполнителя**.
>
> ⚠️ **Облачные сессии видят только свой репозиторий** и до этого файла не дотянутся. Поэтому каждый продукт обязан нести короткую памятку в своём `AGENTS.md` — шаблон: [`templates/AGENTS-stub.md`](templates/AGENTS-stub.md).

## Роли

- **Владелец / архитектор системы — Кирилл (человек).** Направление, продукт, приоритеты, ресурсы, «что и зачем». Принимает продуктовые развилки. Назначает резервного релиз-инженера, когда основной недоступен.
- **Архитектор-исполнитель — Claude Code.** Превращает замысел в структуру; ревью, версии, мерж в `dev`, теги, выкатка; **держит стыки между продуктами**; координирует агентов; отвечает за качество и процесс. **Один на всю экосистему** (не по одному на продукт) и **дефолт-держатель эстафеты релиза**.
- **Рабочая сессия / субагент.** Спавнится под задачу, живёт коротко, **отдаёт результат архитектору-исполнителю**. НЕ мержит в `dev`/`main`, НЕ ставит теги, НЕ деплоит.
- **Воркеры (эксплуатация).** Периодические (синк, отчёты, health-check). **Предлагают, а не исполняют необратимое.** Любой сторонний эффект — только через гейт архитектора-исполнителя.

_Почему архитектор один: локальный архитектор «только своего продукта» не видит стык. Пример: QR рабочих мест физически лежит в БД кабинета, а место ему в админ-платформе — такое решение принимает только тот, кто держит оба конца. Реализация при этом параллелится по продуктам (рабочие сессии), голова по стыкам — одна._

## Релиз-процесс (стандарт)

1. **Источник правды — `dev`.** Релизы — аннотированные теги `vX.Y.Z`. Прод = задеплоенный тег. **Ничего на проде мимо `dev`.**
2. **Прод сам сообщает версию.** `VERSION` пишется при деплое (тег + sha + дата + кто) и отдаётся health-эндпоинтом. «Что на проде» = *спросить прод*, а не гадать.
3. **Журнал релизов = git-теги + `CHANGELOG`** (Keep a Changelog: `[Unreleased]` копится → переносится в версию при выкатке).
4. **Один релиз-инженер в моменте.** Дефолт — Claude. Кончились лимиты — владелец **явно** передаёт эстафету. **Двое одновременно не выкатывают** (иначе гонка тегов — уже ловили).
5. **Протокол «догона»** (в начале релиз-смены / при возврате): спросить версию прода → сверить с тегами и `dev` → проверить миграции → только потом мержить/выкатывать.

## Ветки · версии · миграции

- `feature/<…>` от `dev`; после мержа ветку удалять. Версии **монотонны, не переиспользовать** (SemVer; pre-1.0 → `0.y.z`).
- **Заморозка после хендоффа:** отданную ветку не дописывать; готовые коммиты не переписывать — `amend`, `rebase`, `push --force` **запрещены** (история append-only). Новая доработка = новая ветка/итерация.
- Миграции доходят до прода **только через `dev`**. Конфликт миграций → **merge-нода**, НЕ ручная хирургия боевой БД.

## Жёсткие запреты (для всех, кроме релиз-инженера в его окне)

- Не выкатывать в прод мимо процесса: ssh/docker/deploy-скрипты на боевых серверах.
- Не пушить и не мержить в `main`/`dev`, не ставить теги — это делает только релиз-инженер.
- Не удалять данные/файлы/миграции и не делать необратимое без явного указания владельца.
- Не создавать аккаунты, не менять права/секреты, не подключать платные сервисы, не отправлять наружу.

## Хендофф архитектору (как рабочая сессия заканчивает)

Проверки зелёные (сборка/линт/типы + тесты; красное = не отдавать) → коммит в **свою** ветку → пуш **ветки** (не `dev`/`main`) → запись в `CHANGELOG` `[Unreleased]` → сводка «сделано / проверить / осталось» → **последняя строка**:

```
Архитектору: смержи и выкати <ветка> (commit <полный SHA>)
```

Не готово к проду → `Архитектору: в прод не мержить, ветка в работе`.

_В продуктах, где принят PR-поток (напр. админ-платформа): финал = **Pull Request** + та же строка архитектору. Мерж/деплой — не сессия._

## Что документируем, а что нет (контракт — вперёд, а не всё — вперёд)

Тест один: **«зависит ли от этого кто-то снаружи — другой модуль, другой агент, другой экран?»**

- **Да, пересекает границу → документ = истина**, пишем/обновляем **вместе с кодом** (в том же коммите/PR): границы модулей, схема данных для читающих, контракты API/интеграций, доменные правила. Крупную/рискованную фичу предваряет короткая записка (что и зачем), не полный спек.
- **Нет, живёт внутри одной фичи и ещё лепится → без жёсткого документа.** Истина о поведении = **код + тесты**, не проза. Не документируем каждую функцию (протухнет и станет врать).
- **Переезд.** Застыло и на это обопёрся кто-то ещё → стало контрактом → переезжает в документ.

_«Истина — код» для поведения остаётся в силе; документ — истина для **намерения и контракта**, а не повод ломать рабочий код под устаревший абзац._

## База знаний — слои (растим по нужде, не впрок)

- **L0** — этот устав (роли + процесс + запреты). Общий для экосистемы.
- **L1** — карта архитектуры (границы модулей, где что живёт, источник правды).
- **L2** — домен (модель оплаты, грейды, услуги/опции, YClients).
- **L3** — конвенции (код-стиль, бренд, версии, миграции).
- **L4** — ранбуки (деплой, догон, инцидент) — для воркеров.

_Принцип: не плодить документы впрок (устареют), а консолидировать существующее и добавлять слои по мере реальной нужды._
