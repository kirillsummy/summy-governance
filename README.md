# summy-governance

**Канон экосистемы SUMMY** — роли, процесс разработки, правила релизов, запреты.

## Зачем отдельный репозиторий

Это решения уровня **всей экосистемы и процесса разработки**, а не одного продукта. Раньше устав лежал внутри репо кабинета мастера — и это структурная ошибка: канон экосистемы не должен жить внутри одного из её продуктов.

Она выстрелила 16.07.2026: облачная сессия админ-платформы **не знала, что существует роль архитектора** — устав был в чужом репо, а её собственный `AGENTS.md` описывал только «что нельзя», но не «кто есть кто».

## Что здесь

| Файл | Что |
|---|---|
| [`CHARTER.md`](CHARTER.md) | **Устав (L0)** — роли, продукты, релиз-процесс, ветки/версии/миграции, запреты, хендофф, правило документации. Читают все агенты. |
| [`decisions/`](decisions/) | **Межпродуктовые решения (ADR)** — принятые архитектурные контракты экосистемы. Первое: [платформа данных](decisions/0001-data-platform.md). |
| [`templates/AGENTS-stub.md`](templates/AGENTS-stub.md) | Шаблон **памятки** для `AGENTS.md` каждого продукта. |

## Как это работает

**Облачная сессия видит только свой репозиторий** — до этого репо она не дотянется. Поэтому:

- **Канон — здесь, в одном месте.** Правим только тут.
- **В каждом продукте — короткая памятка** (шаблон выше): кто есть кто, кому отдаёшь работу, чего не делаешь, ссылка сюда. Памятка — **не копия устава**, а минимум + указатель.
- **Специфика продукта остаётся в его репо** (`CLAUDE.md`/`AGENTS.md`): стек, команды, как деплоить именно его.

## Продукты

Приложение мастера (`kirillsummy/SUMMY`, `master.summy.ru`) · Приложение клиента (пока нет) · Админ/операционная платформа (`kirillsummy/adminapp`, `admin.summy.ru`) · Сайт (`summymanager-rgb/summy`, `summy.ru`).

Архитектор — **один на все**: ценность и риск живут на стыках между продуктами.
