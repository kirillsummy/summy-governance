# Контракт: тождество мастера (YClients user ↔ staff)

Как продукты понимают, что «сотрудник в филиале А» и «сотрудник в филиале Б» —
один человек. Читают кабинет, adminapp, платформа данных, clientapp.

## Модель YClients

- **Сотрудник (staff)** — карточка в конкретном филиале: график, услуги,
  записи. У одного человека в N филиалах — N карточек с разными `staff_id`.
- **Пользователь (user)** — учётка входа (телефон), одна на человека.
  Единственная связь карточек между филиалами — **привязка одного и того же
  пользователя** к каждой из них (`staff.user_id`).

## Правила

1. **Ключ тождества — `user_id`.** Карта `{company_id: {user_id: staff_id}}`
   строится по спискам staff всех филиалов студийным токеном; карточки без
   `user_id` в тождество не попадают.
2. **По имени не матчить никогда.** Имена грязные (три «Полины» в одной студии,
   хвостовые пробелы); промах = человек видит чужие записи с именами клиентов.
3. **Вход мастера** = `/auth` (личный логин) → `auth.data.id` (user_id) →
   поиск карточек по всем филиалам студийным токеном. Личный токен мастера
   не сохраняется; права во втором филиале мастеру не нужны.
4. **Второй филиал виден только при привязке**: нет пользователя на карточке →
   продукты честно показывают один филиал, ничего не падает.
5. Операционное правило студии: **выводишь мастера в новый филиал — привяжи
   к его карточке того же пользователя** (тот же телефон). Обратная сторона:
   привязать чужого пользователя = показать человеку чужой график.

Реализация-образец: кабинет `apps/yclients/services.py` —
`build_branch_staff_index`, `master_schedule_identities`.
