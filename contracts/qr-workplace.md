# Контракт: QR рабочего места

**Межпродуктовый контракт** — поэтому лежит в каноне (`summy-governance`), а не внутри
одного из продуктов: облачная сессия видит только свой репозиторий.

- **Генератор** — админ-платформа (`kirillsummy/adminapp`, `admin.summy.ru`).
- **Читатель** — кабинет мастера (`kirillsummy/SUMMY`, `master.summy.ru`); реализовано в v0.25.0.

Общей БД у них нет, поэтому формат ниже — **единственная связь**. Меняется только
согласованно с обеими сторонами: правим здесь, затем обе стороны.

## Payload

```
summy://workplace?salon=<yclients_company_id>&resource=<yclients_resource_id>
```

Пример (SUMMY Батурина, «Маникюрный стол №1»):

```
summy://workplace?salon=481570&resource=101890
```

- `salon` — ID филиала YClients. Кабинет по нему находит студию и её геозону.
  Батурина `481570`, Офицерская `386571`.
- `resource` — ID экземпляра ресурса YClients. **Тип и номер места уже внутри
  ресурса** (`101890` = «Маникюрный стол №1»), отдельных полей не нужно.

## Почему статичные ID, а не токен

QR — это статичные данные о рабочем месте, а не секрет. Поэтому:

- Генератору **не нужна ничья БД**: он собирает payload из тех же ID YClients.
- Обе стороны читают QR независимо, без общих токенов и без синхронизации.
- От подделки защищает **геозона** (открыть смену можно только физически у студии),
  а не секретность payload.

Случайные токены использовать **нельзя**: они требуют совпадения БД двух продуктов.

## Откуда брать список мест (генератор)

```
GET https://api.yclients.com/api/v1/company/{company_id}/resource_instances
```

Ответ — JSON:API-формат **без обычной обёртки `success`**:

```json
{"data": [{"type": "resource_instance", "id": "101890",
           "attributes": {"title": "Маникюрный стол №1"}}]}
```

Пример разбора на стороне кабинета — `YClientsClient.resource_instances()`.

## Подводные камни данных (проверено на боевых)

- **Один физический объект может быть заведён несколькими ресурсами.** У Офицерской
  «Кабинет №3» — это `168690` и `106124` (костыль YClients, чтобы развести услуги).
  Физически кабинет **один**: наклейка **одна**, а кабинет при скане принимает
  **любой** из ID и находит то же место (`Workplace.yclients_resource_ids`).
- **Названия неаккуратные:** «Реклайнер #1» (решётка), «Маникюрный стол №4 »
  (хвостовой пробел). Связывать **только по ID**, номер из названия не парсить.
- Человеческие имена (relabel) — см. `sync_workplaces_from_yclients`:
  гримерное зеркало → «Место бровиста», реклайнер → «Место лешмейкера»
  (реклайнеры = ресницы: ламинирование/наращивание).

## Что делает кабинет при сканировании

1. `salon` → филиал по `yclients_company_id` → проверка геозоны и точности.
2. `resource` → рабочее место по `yclients_resource_ids`.
3. Место пишется в смену (`Shift.workplace`) — всегда видно, кто за каким местом
   работал (разбор поломок и нарушений порядка).

Неизвестный `resource` смену не ломает — она откроется без места.
Реальная QR/гео-проверка включается флагом `SHIFT_QR_REQUIRED=1`.
