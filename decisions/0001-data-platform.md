# ADR-0001 · Платформа данных SUMMY

**Статус:** принято владельцем · 16.07.2026
**Кто решал:** Кирилл (владелец) на сессии проектирования с архитектором. Это решение — межпродуктовый контракт, поэтому живёт в каноне.

## Принцип владельца (отправная точка)

> Единая **своя** база данных, из которой строятся любые отчёты и интерфейсы. Данные хранятся и собираются **нами**, а не сторонними сервисами. Коробочные CRM не ложатся на наши процессы: они задумывались как универсальные механизмы, а механизмы не универсальны.

YClients остаётся источником записей — но не нашей истиной навсегда. Продукты строят отчёты из нашей платформы.

## Контекст на момент решения

- **Кабинет мастера** (`kirillsummy/SUMMY`, Django) уже держит частичное зеркало YClients у себя: синк сегодняшнего дня каждые 5 минут, окна «−1…+14 дней» — каждые 30 минут, плюс сотрудники, графики, зарплатные расчёты. Код проверенный и идемпотентный (upsert по `record_id`, закрытые мастером записи не перезатираются, удалённые помечаются отменёнными). Но хранится только **текущее состояние** в узком окне дат — сырых ответов YClients и истории изменений нет, «любое состояние отчёта задним числом» из этого не построить.
- **adminapp** (`kirillsummy/adminapp`, Next.js) — БД нет, v1 на моках. Управленка (P&L, статьи, журнал) — в Excel.
- **`summy-data-gateway`** — репозиторий создан и пуст. **`clientapp`** — только README.
- **Timeweb Cloud** уже используется (S3 `s3.twcstorage.ru`, регион ru-1).
- Наработка управленческой отчётности — ветка `claude/management-reporting-web-program-wqh25e` в `kirillsummy/SUMMY`: модели, расчёт P&L «до рубля» с тестами, сид-миграция с разбором таблицы «30 — Финансы». Из кабинета откачена (последний коммит ветки — revert): решение «управленке не место в кабинете» уже принято де-факто. Ветка используется как ТЗ.

## Решение

### 1. Где живёт платформа

Отдельная **управляемая Postgres в Timeweb Cloud**, тот же регион, что S3 (ru-1). Бэкапы, обновления и восстановление — на сервисе: данные платформы со временем станут самым ценным активом, их сохранность не должна зависеть от нашей дисциплины бэкапов.

**БД кабинета никуда не выносится** — она остаётся локальной (Docker рядом с Django, доли миллисекунды на запрос). Поэтому латентность до managed-инстанса критична только для гейтвея и adminapp, а они к ней терпимы.

Фаза 0 — проверить задержку от сервера кабинета (`81.200.158.170`) до managed-инстанса. Запасной путь, если задержка окажется неприличной: Postgres в Docker на отдельном VPS (минус — бэкапы снова наши).

### 2. Слои внутри платформенной БД

| Слой | Что | Правило |
|---|---|---|
| `raw` | Каждый ответ YClients как есть (JSONB) + когда получен + хеш. Append-only. | Никогда не удаляем и не правим. История — навсегда; любой будущий отчёт можно построить задним числом. |
| `core` | Нормализованное текущее состояние: записи, клиенты, сотрудники, услуги, филиалы, графики, зарплатные расчёты, кассовые операции + межпродуктовые обогащения. | Удаления мягкие (`deleted_at`). Перестраивается сколько угодно. |
| `contract_v1` | Витрины (view), которые читают продукты. | Стабильны. Ломающее изменение = `contract_v2` рядом и переезд читателей в своём темпе, не правка на месте. |

### 3. Кто пишет

Только **`summy-data-gateway`**. Стек — Django без UI: переносим проверенный код `apps/yclients` кабинета (клиент API, upsert-сервисы, журнал синков) вместо переписывания.

Расписание: сегодняшний день — каждые 5 минут; окно −1…+14 дней — каждые 30 минут; ежесуточный «хвост» на 60–90 дней назад (правки задним числом: посещаемость, возвраты); справочники (сотрудники, услуги, филиалы) — раз в сутки. Разовый **бэкфилл всей истории с открытия студий**. Новое к тому, что кабинет уже умеет: сырьё с историей и **кассовые операции YClients** (для P&L выручка по оплатам важнее, чем по записям).

### 4. Кто и как читает

- Свои продукты (adminapp; позже кабинет): SQL **только к contract-схемам**, каждому продукту — своя read-only роль. Прямое чтение `core`/`raw` запрещено — иначе связанность схем вернётся.
- **clientapp** и всё внешнее: **только через API** поверх contract-слоя. Наружу SQL не выдаём никогда. API строим, когда появится clientapp, — не раньше.
- Продукты пишут только в **свои** БД. В `raw`/`core` пишет только гейтвей.

### 5. Судьба БД кабинета

Остаётся приложенческой. Синк кабинета продолжает работать как сейчас; в переходный период YClients опрашивается дважды (кабинетом и гейтвеем) — при текущих объёмах это далеко от лимитов API. Отдельным решением позже (когда платформа докажет надёжность) кабинет переводится на получение зеркала из платформы, свой синк выключается — одна точка правды, один опрос YClients.

Полный перенос БД кабинета в managed **отвергнут**: сетевой хоп на каждый запрос приложения + простой при миграции — без какой-либо выгоды.

### 6. Управленка (замена Excel)

Продукт — **adminapp**. Статьи, параметры, журнал операций, план выручки — его собственные данные: живут в **его** базе (физически можно на том же managed-инстансе отдельной БД и ролью), пишет в них только adminapp. Факт выручки берётся из витрины платформы, не из кабинета.

**ТЗ** — ветка `claude/management-reporting-web-program-wqh25e`, коммит `4b6410d` (состояние до revert):
- модели `Article` (группы P&L: доходы / себестоимость / переменные / постоянные / амортизация / налоги; норма в % от NET), `Parameter`, `RevenuePlan` (план NET по месяцам), `JournalEntry` (факт по статьям);
- расчёты `pnl_report`, `compute_scenarios`, `yearly_facts`, `subarticle_breakdown` — воспроизводят таблицу «до рубля», тесты зелёные;
- сид-миграция с разбором «30 — Финансы» — конвертируется в сид adminapp.

Ветку **не мержить** — это источник спецификации, не код для выкатки.

## Порядок шагов (ценность — раньше)

| Фаза | Что | Ценность |
|---|---|---|
| 0 | Проверка латентности, создание managed Postgres (тариф подтверждает владелец) | — |
| 1 | Гейтвей v0: `raw`+`core`, перенос синка из кабинета, журнал синков, бэкфилл истории | Полная **своя** копия данных YClients с историей — принцип владельца выполнен |
| 2 | `contract_v1`: витрина выручки/записей по филиалам, мастерам, услугам | Фундамент для управленки и дашборда |
| 3 | adminapp: своя БД + управленка по ТЗ + факт из витрины | **Excel-управленка умирает** |
| 4 | Дашборд adminapp переезжает с моков на витрины | Живые цифры у управляющих |
| 5 | (позже, отдельное решение) кабинет получает зеркало из платформы, свой синк выключается | Одна точка правды |

## Риски — честно

- **Ещё один работающий сервис.** Гейтвею нужен присмотр: журнал синков (переносится с кодом) + health-эндпоинт по уставу.
- **Двойной опрос YClients** до фазы 5 — приемлемо при текущих объёмах; следить за лимитами API.
- **Дисциплина контрактов.** Соблазн «по-быстрому» прочитать `core` напрямую вернёт связанность. Правило зафиксировано этим документом.
- **Managed БД — ежемесячные деньги.** Тариф — за владельцем.
- **Расхождения цифр** кабинета и платформы до фазы 5 (два независимых синка) — возможны и объяснимы; сверка по `record_id`.
