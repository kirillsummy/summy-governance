# Шаблон памятки для `AGENTS.md` продукта

Вставь блок ниже в `AGENTS.md` (или `CLAUDE.md`) **каждого** продукта экосистемы, ближе к началу.

**Зачем:** облачная сессия видит **только свой репозиторий** и до общего устава не дотянется. Памятка — не копия устава, а минимум, чтобы сессия, прочитавшая только свой репо, знала кто есть кто и кому отдаёт работу, плюс указатель на канон.

**Правило:** памятка короткая и меняется редко; канон правим **в одном месте** — `summy-governance/CHARTER.md`.

---

```markdown
## Роли и процесс (канон экосистемы SUMMY)

> Полный устав — **[`summy-governance/CHARTER.md`](https://github.com/kirillsummy/summy-governance/blob/main/CHARTER.md)**. Здесь — минимум, потому что облачная сессия видит только свой репозиторий.

- **Кирилл — владелец / архитектор системы** (человек): направление, продукт, приоритеты, продуктовые развилки.
- **Claude Code — архитектор-исполнитель**: ревью, версии, мерж в `dev`, теги, выкатка, **стыки между продуктами**, координация агентов. **Один на всю экосистему** (не по одному на продукт).
- **Ты — рабочая сессия**: делаешь фичу и **отдаёшь архитектору**. Сама НЕ мержишь в `dev`/`main`, НЕ ставишь теги, НЕ деплоишь.

**Экосистема — 4 продукта, и у каждого свои границы:**

| Продукт | Что тут строим |
|---|---|
| Приложение мастера (кабинет, `master.summy.ru`) | Всё про мастера: смены, записи дня, его финансы/тариф, профиль, портфолио |
| Админ / операционная платформа (`admin.summy.ru`) | **Вся управленка**: дашборд управляющего, операционные процессы, отчёты студии, QR-генератор, «Доработки». Сюда переезжает то, что жило в Excel |
| Сайт (`summy.ru`) | Публичное: лендинги, SEO, блог, вакансии |
| Приложение клиента (пока нет) | Клиентское |
| Платформа данных (`summy-data`, план) | API-шлюз + наша БД; пишет только шлюз, продукты читают contract-слой |

**Правило границ:** прежде чем строить — **определи, чей это продукт**. Строишь не в своём — **остановись и скажи**. Не тащи чужой функционал в свой репозиторий только потому, что ты в нём сидишь. Задача упирается в чужой продукт (его данные, API) — **не решай за него**, опиши развилку в PR/сводке.

**Межпродуктовые контракты** (читают ≥2 продукта) живут в каноне `summy-governance/contracts/`, а не внутри продукта — например формат QR рабочего места. Свой репозиторий — не единственный источник правды.

**Как заканчиваешь работу:** проверки зелёные → коммит в СВОЮ ветку → пуш ветки (не `dev`/`main`) → запись в `CHANGELOG` → сводка «сделано / проверить / осталось» → последней строкой:

    Архитектору: смержи и выкати <ветка> (commit <полный SHA>)

Не готово к проду → `Архитектору: в прод не мержить, ветка в работе`.

**История append-only:** отданную ветку не дописывай; `amend`/`rebase`/`push --force` по готовым коммитам запрещены. Новая доработка = новая ветка.
```
